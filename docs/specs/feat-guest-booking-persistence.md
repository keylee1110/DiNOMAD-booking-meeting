# Spec: Booking Persistence to Database

**Status:** `done`
**PRD ref:** §5.4 (Payment Success), §6 (Booking Confirmation), §7 (My Bookings)
**Branch:** `main`
**Author:** Nguyen Dieu Vi
**Date:** 2026-06-20

---

## Overview

Confirmed bookings were not being saved to the database for authenticated users. The checkout flow inserted directly into Supabase but sent a human-readable id (`BK-20260620-042`) into the `bookings.id` column, which is a `uuid`. Postgres rejected every insert; the error was shown only as a brief toast and the flow fell through to localStorage and redirected to the success page anyway — so bookings *looked* confirmed while no DB row existed. This spec documents the fix that makes bookings persist to Supabase for logged-in customers.

---

## Acceptance Criteria

- [x] An authenticated customer completing checkout creates a real row in `public.bookings`
- [x] The DB row uses a server-generated `uuid` primary key (not the `BK-...` string)
- [x] The unique `booking_code` (`DN-XXXXXX`) is generated by the DB trigger and read back into the client booking object
- [x] The success page, My Bookings, and a hard refresh all resolve the booking from the DB by its real UUID
- [x] If an authenticated insert fails, the flow surfaces the real error and stops — it no longer fakes success
- [x] Guests (unauthenticated) and mock/non-UUID rooms still fall back to the localStorage cache unchanged

**Out of scope for this spec:**
- Guest booking persistence (no `customer_id` to satisfy RLS) — deferred
- Real PayOS payment integration (still simulated) — deferred to a payments spec
- Server-side slot-conflict validation via a dedicated `POST /api/bookings` backend endpoint — deferred

## Backend

No new backend endpoints. Booking creation uses a **direct Supabase insert from the client** (same pattern as wishlist/points), governed by RLS.

### DB changes

No new migrations. The fix relies on already-applied schema in
`supabase/migrations/20260605000000_bookings_payments_reviews.sql` and
`supabase/migrations/20260605010000_profile_points_booking_code_wishlists.sql`:

- `bookings.id` — `uuid primary key default gen_random_uuid()` (client must NOT supply it)
- `booking_code` — populated by trigger `bookings_generate_booking_code` when null
- RLS insert policy `"Customers can create bookings"` requires `customer_id = auth.uid()`

### Service logic

```
1. Build UTC-safe start/end timestamps from selectedDate + slot range (treat as UTC+7)
2. Insert payload WITHOUT id / booking_code → let DB generate both
3. .select().single() to read back data.id (uuid) + data.booking_code
4. On success → use returned values downstream
5. On error (authenticated) → toast real message, reset paying state, return (no fake success)
```

### Error cases

| Condition | Behaviour |
|---|---|
| Authenticated insert fails (RLS, constraint, overlap) | Toast real error, abort — stays on checkout |
| Guest (no user) | Skip DB, save to localStorage only |
| Mock room (non-UUID id) | Skip DB, save to localStorage only |

## Notes

- **Root cause:** `generateBookingId()` returns `BK-{date}-{rand}`, which is not a valid UUID. Passing it as `bookings.id` failed the `uuid` column type check on every authenticated insert.
- **Decision:** Let the database own identity generation (`gen_random_uuid()` + booking-code trigger) rather than the client. The client now treats the insert result as the source of truth.
- **Follow-ups:**
  - Guest booking persistence (phone-based identity or guest profiles) — requires RLS/auth design.
  - A proper `POST /api/bookings` NestJS endpoint for server-side slot-conflict validation and the 5-minute soft lock (PRD §4.3).
  - Real PayOS payment instead of the simulator modal (PRD §5).
